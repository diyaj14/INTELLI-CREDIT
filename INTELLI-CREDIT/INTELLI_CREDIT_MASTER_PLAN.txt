────────────────────────────────────────────────────────────────────────────────
1.1 BANKING & CREDIT DOMAIN KNOWLEDGE  (Days 1–2)
────────────────────────────────────────────────────────────────────────────────

A. The Indian Corporate Credit Lifecycle
   - Loan Application → Document Collection → Credit Analysis → CAM Preparation
     → Credit Committee → Disbursement
   - Why it takes 3 weeks: 40+ documents from 8+ sources, manual cross-linking
   - Search: "corporate loan appraisal process India RBI guidelines"
   - Find a real bank CAM template: Search "SBI credit appraisal manual PDF"
     or "PNB CAM format India" — reading ONE real CAM teaches more than videos

B. The Five Cs of Credit (in Indian Context) — MUST KNOW COLD
   - CHARACTER   → CIBIL Commercial report, MCA director history, court records
   - CAPACITY    → DSCR (Debt Service Coverage Ratio), current ratio, cash flows
                   Formula: DSCR = Net Operating Income / Total Debt Service
   - CAPITAL     → Net Worth, Debt-to-Equity ratio, promoter skin in the game
   - COLLATERAL  → Fixed Asset Register, property valuations, MCA charge creation
   - CONDITIONS  → Industry headwinds, RBI sector circulars, NPA data for sector

C. Key Financial Ratios You Must Understand:
   - Current Ratio (CR): Current Assets / Current Liabilities (ideal: >1.33)
   - Debt to Equity (D/E): Total Debt / Shareholders' Equity (ideal: <2:1)
   - DSCR: EBITDA / (Principal + Interest) (ideal: >1.25)
   - Interest Coverage Ratio: EBIT / Interest Expense (ideal: >3)
   - Net Profit Margin, Gross Profit Margin, Return on Equity

D. Indian Regulatory Framework
   - RBI Master Circular on Loan System for Delivery of Bank Credit
   - RBI Early Warning Signals (EWS) guidelines for NPA detection
     Search: "RBI early warning signals NPA framework circular"
   - SARFAESI Act basics (how banks recover loans from defaulters)
   - Insolvency and Bankruptcy Code (IBC) — what happens when a company defaults
   - CIBIL Commercial Score vs CIBIL Rank — understand both

────────────────────────────────────────────────────────────────────────────────
1.2 GST INTELLIGENCE (Critical Differentiator) (Day 2–3)
────────────────────────────────────────────────────────────────────────────────

   This is your BIGGEST DIFFERENTIATOR. Judges will probe this deeply.

A. GST Return Types — Know the Difference:
   - GSTR-1:  Monthly outward sales return (what the company SOLD)
   - GSTR-3B: Self-assessed monthly summary return (what company REPORTS)
   - GSTR-2A: Auto-populated inward purchases from supplier GSTR-1 filings
   - GSTR-2B: Static monthly Input Tax Credit (ITC) statement

B. The Fraud Signals You Must Detect:
   - GSTR-3B vs GSTR-2A MISMATCH: If claimed ITC > what suppliers filed,
     someone is claiming fake input credits → fraud red flag
   - GSTR-1 vs Bank Statement Mismatch: If declared sales ≠ actual bank credits,
     revenue is being inflated → credit risk
   - Circular Trading: Company A sells to B, B sells to C, C sells back to A —
     creates fake turnover. Detectable via cross-GST analysis.
   - Search: "GSTR-2A vs GSTR-3B mismatch fraud signal India fintech"
   - Search: "ITC fraud India GST circular trading detection method"

C. Who to Study:
   - Perfios, CredAble, Karza Technologies — Indian fintechs doing this commercially
   - Account Aggregator (AA) framework by RBI — consent-based bank data sharing
   - Sahamati AA ecosystem — your architecture should reference this

────────────────────────────────────────────────────────────────────────────────
1.3 MCA (Ministry of Corporate Affairs) DATA (Day 3)
────────────────────────────────────────────────────────────────────────────────

   - Go to mca.gov.in → search any company name → understand what you see
   - MCA Master Data: CIN (Company Identification Number), registered address,
     date of incorporation, status (Active/Struck Off), paid-up capital
   - DIN (Director Identification Number): links directors across all companies
   - MCA Charges: When a bank takes security on company assets, it registers a
     "charge" on MCA. Outstanding charge = existing bank loan! Critical signal.
   - ROC Filings: Annual Returns (MGT-7), Financial Statements (AOC-4)
   - Why this matters: A "clean" company on paper might have struck-off
     subsidiaries or directors linked to fraud cases. Your agent must find this.

────────────────────────────────────────────────────────────────────────────────
1.4 AI/ML & DATA ENGINEERING FOR CREDIT (Days 4–6)
────────────────────────────────────────────────────────────────────────────────

A. Document Processing Reality for Indian PDFs:
   - Indian CA documents: scanned at wrong angles, mixed Hindi/English,
     tables within tables, handwritten annotations, low-quality scans
   - Learn: pdfplumber vs Camelot vs PyMuPDF vs Unstructured.io
   - OCR: Tesseract (free) vs PaddleOCR (better for Indian docs) vs Azure DI
   - Critical: Understand WHY naive PDF parsing fails on financial tables

B. RAG (Retrieval-Augmented Generation):
   - Core pattern: Ingest docs → chunk → embed → store in vector DB
     → user query → retrieve relevant chunks → LLM synthesizes answer
   - LangChain or LlamaIndex as the orchestration layer
   - ChromaDB or FAISS as the vector store
   - Key limitation to know: Naive RAG fails on numerical tables — learn
     "table-aware chunking" and "structured extraction before RAG"

C. Agentic AI (ReAct Pattern):
   - Research Agent needs: Search → Read → Decide if relevant → Search more
   - Different from RAG: Agent actively DECIDES what to look for
   - LangChain agents with tools: web search (Tavily), MCA scraper, e-Courts
   - Search: "ReAct agent pattern LLM LangChain tutorial 2024"

D. Explainable AI (XAI) — THE Most Important Technical Concept:
   - Problem statement EXPLICITLY demands explainability
   - Learn SHAP values: how each feature contributes to the credit score
   - Learn LIME: Local Interpretable Model-agnostic Explanations
   - Best approach for judges: LLM-generated natural language explanations
     layered ON TOP of a traditional scorecard model
   - "This company was rejected because: GST mismatch (−35 pts), litigation
     risk found in secondary research (−28 pts) despite strong bank flows (+40 pts)"

E. Databricks (Problem Statement Mentions This Explicitly):
   - Know: Delta Lake (their data format — think "data lakehouse")
   - Know: MLflow (experiment tracking, model versioning)
   - Know: Databricks Notebooks (Spark-based collaborative environment)
   - You DO NOT need to use Databricks in the prototype — but USE THE TERMINOLOGY
   - Your architecture diagram should show "Databricks Delta Lake" as the
     data storage layer. This shows you read the brief. Judges notice.

────────────────────────────────────────────────────────────────────────────────
3.1 TECHNICAL CHALLENGES
────────────────────────────────────────────────────────────────────────────────

CHALLENGE 1: Indian PDF Parsing Quality
  Problem:   Real Indian CA-certified documents are often scanned, rotated,
             have handwritten notes and mixed Hindi/English text
  Risk:      OCR errors cascade → wrong financial numbers → wrong credit decision
  Mitigation:
    - Use PaddleOCR (better than Tesseract for Indian scripts)
    - Manual validation step: show extracted values to user for confirmation
    - Build confidence scoring: "Extracted with 94% confidence"
    - Curate a test set of 3–5 representative Indian PDFs for your demo

CHALLENGE 2: LLM Hallucination on Financial Data
  Problem:   LLMs tend to confabulate numbers they cannot read clearly
  Risk:      Wrong revenue figure → wrong credit limit → liability in real world
  Mitigation:
    - Never trust LLM for number extraction — use deterministic parsing first
    - LLM's role: INTERPRET numbers, not EXTRACT them
    - Show source citations in the CAM: "Revenue ₹42 Cr (Source: PL Page 3, Line 12)"

CHALLENGE 3: Real-Time Web Scraping Reliability
  Problem:   mca.gov.in and ecourts.gov.in are slow, change layout frequently,
             and have CAPTCHAs
  Risk:      Research agent fails during live demo
  Mitigation:
    - Cache all web results locally during the demo
    - Have 2–3 "pre-researched" companies whose data is already cached
    - Show the scraping in a non-blocking async queue — UI continues working
    - Backup: Use pre-fetched JSON as "live" data if scrapers fail

CHALLENGE 4: GST Cross-Validation Without Official API Access
  Problem:   Official GSTR APIs require taxpayer consent and GSTIN registration
  Risk:      Cannot do real GSTR-2A vs 3B mismatch analysis in prototype
  Mitigation:
    - Build the cross-validation LOGIC on sample GST Excel exports
    - Frame it as "When connected to GSTN via Account Aggregator, this rule runs"
    - Show the mismatch detection working on synthetic/sample data in the demo

CHALLENGE 5: Explainability vs Accuracy Trade-off
  Problem:   Black-box models (XGBoost, neural nets) have higher accuracy but
             banking regulations require interpretability
  Risk:      Judges might push on why you used a "simple" logistic regression
  Mitigation:
    - PROACTIVELY explain: "We chose a transparent scorecard because RBI
      Circular RBI/2021-22/112 requires credit decisions to be explainable"
    - Show SHAP waterfall chart in the demo — it looks impressive even for LR
    - This is actually your STRENGTH, not a weakness. Own it.

────────────────────────────────────────────────────────────────────────────────
3.2 DOMAIN / KNOWLEDGE CHALLENGES
────────────────────────────────────────────────────────────────────────────────

CHALLENGE 6: Judges Are Bankers, Not Engineers
  Risk:      Technical jargon goes over their heads; domain errors destroy trust
  Mitigation:
    - Every technical component must be explained in banking language first
    - Bad: "We use cosine similarity in our embedding space"
    - Good: "We automatically cross-reference GST returns against bank statements
            the way a senior credit manager would, but in 2 minutes"

CHALLENGE 7: Indian-Context Sensitivity (Explicit Evaluation Criterion)
  Risk:      Using US-style credit models that ignore GSTIN, CIBIL Commercial,
             MCA charges, e-Courts, and Indian accounting standards (Ind AS)
  Mitigation:
    - Make EVERY Indian-specific data source visible in the demo
    - Explicitly name: GSTN, CIBIL Commercial Score, MCA V3, e-Courts,
      SARFAESI, RBI EWS framework — use these terms naturally
    - Indian rupee amounts: always show in crores (₹X Cr), not lakhs
